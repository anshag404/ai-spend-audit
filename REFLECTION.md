# Reflection

## 1. The hardest bug you hit this week, and how you debugged it
The most challenging bug I encountered was a subtle hydration mismatch in the results dashboard. Because the application reads audit data from `localStorage`, there was a split second where the server-rendered HTML (which had no access to `localStorage`) didn't match the client-rendered state. This caused React to throw errors and, more importantly, caused the "Savings Hero" component to flicker or show $0 before snapping to the correct value. 

My initial hypothesis was that I could just check for `window` availability, but that still led to content jumping. I tried several approaches, including putting the `localStorage` logic in a `useLayoutEffect`, but that didn't solve the server/client mismatch. Finally, I realized I needed a "Two-Pass Rendering" strategy. I implemented a `DashboardState` that explicitly starts as `loading`. I then used a `useEffect` to trigger the load from `localStorage` only after the component had mounted on the client. To prevent the "flash of empty content," I built a `DashboardSkeleton` that mirrors the layout of the final dashboard. This ensured that the server and client both rendered the same skeleton initially, and the real data hydrated gracefully without a single React warning. This debugging process taught me that when building no-auth, client-side-first apps, you must treat your local state as an "external store" that requires careful synchronization with the React lifecycle.

## 2. A decision you reversed mid-week, and what made you reverse it
Mid-week, I originally planned to use an LLM (specifically Gemini) to generate the actual savings recommendations. I thought it would be "smarter" and could handle more complex natural language inputs. However, after prototyping this on Day 3, I quickly reversed the decision. I found that even with strict system prompting, the LLM would occasionally hallucinate pricing plans that didn't exist or fail to do basic arithmetic correctly (e.g., subtracting $20 from $100 and getting $70).

For a tool that provides financial advice, this level of unpredictability was unacceptable. I realized that "trust" is the most important asset for a SaaS tool. If a user catches the tool in a single math error, they will discount the entire report. I decided to pivot to a deterministic, rule-based engine written in TypeScript. By coding every optimization as a pure function, I could guarantee 100% accuracy and make every recommendation auditable. I relegated the AI to a purely cosmetic role — generating a narrative executive summary *based* on the deterministic data. This reversal actually made the code cleaner and the product much faster, as we no longer had to wait for multiple LLM calls to generate the core report. It was a classic case of "boring code" being superior to "shiny AI" for the core value proposition.

## 3. What you would build in week 2 if you had it
If I had a second week, my primary focus would shift from "audit" to "monitoring." Currently, the tool is a point-in-time snapshot of spend. In week 2, I would implement a "Continuous Optimization" engine. This would involve adding User Authentication (via Supabase Auth) so teams could save their stack and receive monthly alerts. If a tool like Cursor or OpenAI changes its pricing, the system would automatically re-run the audit and send a Slack alert: "Your AI spend just became suboptimal. Switch to the new Team tier to save $200/mo."

Additionally, I would build a "Benchmarking" feature. By aggregating anonymized data from all audits, we could tell a startup: "You are in the 90th percentile for AI spend among seed-stage engineering teams of your size." This creates a powerful FOMO-driven growth loop and provides CFOs with the comparative data they need to justify budgets. I would also add a "Browser Extension" that can detect AI tool usage patterns directly from browser history (with user permission). This would eliminate the manual data entry step, which is currently the biggest friction point in the user journey. By moving from a manual calculator to an automated monitoring platform, the product transitions from a "one-off utility" to a "must-have financial control."

## 4. How you used AI tools
Throughout the build, I used Gemini 1.5 Pro primarily for two tasks: scaffolding complex CSS layouts and generating the initial pricing catalog from unstructured documentation. For the design pass, I gave the AI my color tokens and asked it to "Generate a modern, glassmorphic card component using Tailwind v4 that feels like a premium SaaS dashboard." This saved hours of manual CSS tweaking and resulted in a much higher-fidelity UI than I would have built alone.

However, I did not trust the AI with the core financial math or the database schema design. I caught the AI making a significant mistake when I asked it to write a rule for "prorated annual savings." It failed to account for the fact that some vendors don't offer annual billing for specific tiers. I had to manually correct the logic to include a "hasAnnualOption" check in the tool schema. This reinforced my belief that AI is an incredible "copilot" for aesthetics and boilerplate, but the human must remain the "pilot" for business logic and data integrity. I also used AI to help brainstorm the GTM strategy, which gave me some great ideas for specific subreddits (like r/SaaS) that I hadn't initially considered.
